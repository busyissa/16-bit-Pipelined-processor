# 16-bit RISC Pipelined Processor (Verilog)

This folder contains a 16-bit RISC-style CPU implemented as a simple 5-stage pipeline:

1. IF  (Instruction Fetch)
2. ID  (Instruction Decode / Register Read)
3. EX  (Execute / Address Calc / Branch Resolve)
4. MEM (Data Memory)
5. WB  (Writeback)

The design includes basic forwarding and hazard handling (load-use stall and branch flush).

## Supported ISA (same encoding style as the single-cycle version)

Operations supported by the controller:

- R-type: `ADD`, `SUB`, `AND`, `OR`
- Memory: `LW`, `SW`
- Branch: `BEQ`, `BNE`

Register file is 4 registers: `r0`-`r3` (2-bit register indices). `r0` is hard-wired to `0`.

## Instruction format

Each instruction is 16 bits:

```
[15:10] imm6 | [9:8] rs2 | [7:6] rs1 | [5:4] rd | [3:2] funct2 | [1:0] opcode
```

- `imm6` is sign-extended to 16 bits by `immExt.v`
- The controller decodes `instr[3:0]` (`funct2:opcode`)

## Pipeline structure (what each stage does)

- IF:
  - Fetches `instrF` from `instrMemory.v` using `pcF`
  - Computes `pcPlus1F = pcF + 1`
  - Selects `pcNext` via a mux:
    - normal: `pcPlus1F`
    - taken branch: `pcPlusImmE` (resolved in EX)
  - `pc.v` has an enable input; the hazard unit can stall the PC update

- IF/ID (`fetchReg.v`):
  - Latches `instrF` and `pcF` into `instrD` and `pcD`
  - Can be stalled (hold state) or cleared (flushed)

- ID:
  - Reads register operands from `regfile.v`
  - Sign-extends the immediate
  - Control signals are produced by `controller.v` and are carried forward in the pipeline

- ID/EX (`decodeReg.v`):
  - Latches control + operands + register numbers into EX stage signals
  - Can be cleared to inject a bubble (used for hazards/flushes)

- EX:
  - ALU ops and address calculation
  - Branch compare is done using a SUB-based zero flag in `alu.v`
    - `BEQ` sets the zero flag when `(rs1 - rs2) == 0`
    - `BNE` sets the zero flag when `(rs1 - rs2) != 0`
  - Branch decision signal `branchE` is generated by `andGate.v` when `opcode == 2'b11` and the ALU's branch-appropriate zero flag is true
  - Forwarding muxes (`mux3.v`) can source ALU inputs from:
    - the ID/EX register values
    - WB result
    - MEM ALU result

- EX/MEM (`executeReg.v`):
  - Latches ALU result, store data, destination register, and control bits into MEM stage

- MEM:
  - Loads/stores in `dataMemory.v`
  - Memory indexing is `RAM[aluOut[15:1]]` (halfword-aligned addressing)

- MEM/WB (`memReg.v`):
  - Latches data for writeback

- WB:
  - `resultMux` selects ALU result vs memory read data
  - Writes back to `regfile.v` when `regWrite` is asserted

## Hazard and forwarding logic

`hazard.v` provides:

- Forwarding control (`forwardAE`, `forwardBE`) for EX stage ALU inputs from MEM/WB
- Load-use stall:
  - Detects when an instruction in EX is a load and the instruction in ID needs that result
  - Asserts `stallF` and `stallD` (holds PC and IF/ID)
  - Asserts `flushE` (inserts a bubble into ID/EX)
- Branch flush:
  - When a branch is taken (`branchE`), asserts `flushD` and `flushE` to remove wrong-path instructions

## Key files

- `top.v`: ties together `dp`, `controller`, and `hazard`
- `dp.v`: the pipelined datapath (IF/ID/EX/MEM/WB)
- `controller.v`: control decode (`we3`, `aluMUX`, `aluOp`, `memWrite`, `resultMUX`)
- `hazard.v`: forwarding + stall/flush generation
- Pipeline registers: `fetchReg.v`, `decodeReg.v`, `executeReg.v`, `memReg.v`
- Memories: `instrMemory.v`, `dataMemory.v`
- Core blocks: `pc.v`, `regfile.v`, `alu.v`, `immExt.v`, `adder.v`, `mux2.v`, `mux3.v`, `andGate.v`
- `pipelineTB.v`: simple simulation testbench

## Simulation

If you have Icarus Verilog installed:

```sh
iverilog -g2012 -o sim pipelineTB.v top.v dp.v controller.v hazard.v andGate.v pc.v regfile.v alu.v immExt.v instrMemory.v dataMemory.v fetchReg.v decodeReg.v executeReg.v memReg.v adder.v mux2.v mux3.v
vvp sim
```

ModelSim/Questa-style:

```tcl
vlog *.v
vsim pipelineTB
run -all
```

## Changing the program

The program is hard-coded in `instrMemory.v` inside an `initial begin ... end` block.

To load from a file instead, you can replace the hard-coded program with:

```verilog
initial $readmemh("machinecode.mem", RAM);
```

and create a `machinecode.mem` file containing one 16-bit hex instruction per line.
